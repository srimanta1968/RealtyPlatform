1 · TL;DR

One git repository. Many deployable artifacts. Many databases. A single monorepo holds every microservice, every agent, every frontend, and every shared package. Each microservice still ships its own Docker image, owns its own PostgreSQL database, and deploys independently — but shared types, SDKs, and the design system live in internal packages that everything consumes via workspace references.

2 · Why Monorepo (vs Polyrepo)

Polyrepo is the right choice when you have multiple independent teams, mixed tech stacks, or external consumers of one component. None apply to Kiana.

ConcernPolyrepoMonorepo (Turborepo)
Independent deploys✅✅ via --filter=...[origin/main]
Shared event typesnpm publish + version driftimport from @kiana/contracts
Cross-service refactorN PRs, N reviews, N deploys1 atomic PR
OnboardingClone N repospnpm install
Phase 7 vertical extractionMulti-month migrationFolder restructure
Repo overheadN × CI / READMEs / IaCOne set
Team size to justify20+ engineersWorks from 2 engineers

9.2 What's auto-generated vs hand-written

80% Auto-generated
  • HTTP request/response shapes
  • DTO types
  • Endpoint method signatures
  • Enum values
  • Error response types
20% Hand-written
  • React Query keys + invalidation
  • Optimistic update helpers
  • Cross-service helpers (e.g., leads.scheduleVisit())
  • Typed error classes
  • Pagination iterators
  • Realtime subscriptions (SSE/WebSocket)
  • Upload orchestration
  • Offline queue / sync helpers

10.3 Build configuration (tsup)

// packages/sdk/tsup.config.ts
import { defineConfig } from 'tsup'

export default defineConfig({
  entry: {
    index:     'src/index.ts',
    react:     'src/react/index.ts',
    next:      'src/next/index.ts',
    upload:    'src/upload/index.ts',
    sync:      'src/sync/index.ts',
    concierge: 'src/concierge/index.ts',
  },
  format:    ['esm', 'cjs'],
  dts:       true,
  sourcemap: true,
  splitting: true,
  treeshake: true,
  external:  ['react', 'react-dom', 'next', '@tanstack/react-query'],
  clean:     true,
})

This produces multiple entry points so consumers tree-shake to what they actually use. The Customer Portal pulls only @kiana/sdk + /react + /next; the Field App pulls + /upload + /sync; the public site pulls + /next.

10.4 Versioning & publishing

AspectStrategy
VersioningSingle SDK version, semver. Breaking changes bump major.
Source of truthpackages/sdk/CHANGELOG.md (managed by Changesets)
PublishingWorkspace-only — never published to public npm. Apps reference "@kiana/sdk": "workspace:*".
Internal registryOptional Verdaccio for downstream tools that need built artifacts
Breaking change detectionSpectral rules + API extractor diff in CI
DistributionVia monorepo workspace symlinks. Each app build resolves the current source.

Contract tests guarantee the SDK matches what the service actually exposes — they run against a mock generated from the same openapi.yaml. When a service author changes the OpenAPI in a breaking way, the contract test fails and CI blocks the merge.

10.6 Single-package vs multi-package decision

Single @kiana/sdk

12 · Database Strategy

12.1 Logical isolation, physical consolidation

AspectStrategy
Logical DBsOne per service (lead_db, property_db, ...)
Physical clusters (P1–P3)One managed Postgres cluster, ~10 logical DBs
Physical clusters (P4–P5)Same + read replica for analytics
Physical clusters (P6+)Optionally split heavy services to dedicated clusters
Cross-service joinsForbidden. Use APIs or events.
MigrationsPer service, in db/<service>/migrations/, run as k8s pre-deploy Job

15 · Scalability Path

The structure scales without restructuring. What changes per stage:

StageLead/dayWhat changes
MVP (P1)<100Single Postgres, single Redis, one k8s cluster, 3 nodes
Growth (P3)100–1kAdd read replicas, scale services horizontally, add Kafka if Redis Streams strain

18 · Decisions: Now vs Defer

Decide now
  • Monorepo (Turborepo + pnpm)
  • Folder layout above
  • One Postgres cluster, DB-per-service
  • Node.js + TypeScript + Fastify
  • Drizzle ORM
  • OpenAPI-first → generated SDK
  • Feature-flag tooling
Defer
  • Kafka vs Redis Streams (start Redis, decide P3)
  • Vercel vs self-hosted Next.js (start Vercel)
  • Single vs multi-region (P7 concern)
  • Specific LLM provider
  • Service mesh (not needed until P5+)
  • Separate clusters per environment
  • Splitting SDK into multiple packages (defer until P4)
