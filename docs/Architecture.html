1 · Technology Stack

The platform is intentionally narrow on technology choices to keep operability simple. Three pillars carry the bulk of the system; the rest are deliberately replaceable behind interfaces.

PostgreSQL 16 (per-service schema, pgvector for embeddings) Next.js 14+ (App Router, public site & customer portal) React 18+ (admin cockpit & field PWA) Node.js / TypeScript microservices Temporal (workflow engine) Redis Streams → Kafka (event bus, scale path) S3-compatible object store (media, documents) AI Gateway (Claude / OpenAI / Whisper)
Database
PostgreSQL 16
One physical cluster, one logical database per microservice. pgvector for embeddings (Property Matching). Logical replication into a warehouse for analytics.
Customer Surfaces
Next.js (App Router)
SSR/ISR for the public website (SEO-critical) and the authenticated Customer Portal. React Server Components for fast first paint; Edge runtime where applicable.
Operator Surfaces
React.js SPA + PWA
Admin Cockpit as a Vite + React SPA optimized for dense data. Field App as an installable React PWA with offline queue and background sync.
Services
Node.js / TypeScript
Fastify or NestJS microservices. Shared TS types via an internal @kiana/contracts package. OpenAPI generated for every service.
Orchestration
Temporal + Event Bus
Long-running flows (visit lifecycle, drip campaigns) on Temporal. Fire-and-forget events on Redis Streams initially; scale path to Kafka.
AI
AI Gateway
Provider-agnostic adapter (Claude, OpenAI) with prompt registry, PII scrubbing, cost telemetry, and per-agent budgets.

2 · Layered Architecture

The platform separates five concerns: the customer surface, the operator surface, the AI agent layer that observes events and acts through the same service APIs a human would, the core microservices, and the data + integration plane beneath them.

Five-layer view. The agent layer is deliberately not a service — it subscribes to events and invokes the same APIs operators use.

Why this split

Next.js is chosen for surfaces where SEO, SSR, and first-paint matter (public site, customer portal). React SPA/PWA is chosen for surfaces where data density, interactivity, and offline behavior matter more than SEO (cockpit, field app, marketing console). Both are React under the hood — the design system, types, and SDK are shared, so a developer works seamlessly across them.

Data Strategy Notes

Isolation
No cross-service joins
Services talk via APIs and events, never via shared tables. Data needed elsewhere is duplicated through events with eventual consistency.
Vectors
pgvector inside Property
Embeddings stored alongside property rows. Avoids a separate vector DB until scale demands it (cosine similarity on 1536-dim embeddings is fine into the millions).
Analytics
Logical replication → warehouse
CDC into BigQuery / ClickHouse / Snowflake. Dashboards never query OLTP. Event log is the secondary source of truth for analytical history.
Migrations
Per-service, expand–contract
Each service owns its migrations (Drizzle / Prisma / Knex). Backward-compatible deploy first, code switch second, drop-old third.

7 · AI Agent Layer

Agents are not services — they are event-subscribed handlers running inside the Agent Runtime. They use the same APIs operators use, which makes their actions reviewable, replayable, and reversible.

Phase 3
Lead Qualification
LLM extracts intent / budget / location / urgency from the enquiry. Confidence threshold gates auto-stage moves; below threshold goes to a human review queue.
Phase 3
Campaign Scoring
Hybrid rule + ML scoring on engagement signals. Pushes warm leads to the call queue; explainable components shown in cockpit.
Phase 4
Property Matching
pgvector kNN + rule re-rank + optional LLM reasoning. Solves "Karjat enquiry → Lonavala villa" with explanations.
Phase 4
Visit Summary
Whisper transcribes voice memo; LLM extracts sentiment, objections, interest level, next step. Human confirms with one tap.
Phase 6
Concierge
24/7 inbound chat. Cites Property Service facts only; hands off to humans on low confidence or high-value signal with a structured summary.
Phase 6
Next-Best-Action (LLM)
Synthesizes lead history + recent context to suggest the next action with timing and channel. Surfaces as a task; never auto-executes.

AI Gateway

All agents call the AI Gateway instead of provider SDKs directly. The gateway provides:

  • Provider-agnostic routing with route_hint = quality | latency | cost
  • Versioned prompt registry with rollout gates
  • PII scrubbing before egress
  • Per-agent cost telemetry and monthly budgets
  • Provider failover (primary → secondary) on errors
  • Response caching for static prompts

EKS / GKE. Each microservice deploys independently (Helm). HPA on RPS & queue depth. Next.js can run on Vercel or self-hosted in the cluster.

RDS / Cloud SQL. Read replica for analytics readers. Logical replication slot per service for CDC into the warehouse.
Networking
Edge CDN + WAF
CloudFront / Cloudflare in front of Next.js and the API gateway. WAF rules block obvious abuse; bot challenge on lead capture endpoints.
Observability

10 · Architecture Decisions (ADRs in brief)

#DecisionRationaleTrade-off
ADR-001PostgreSQL per service (one cluster, multi-DB)Strong service boundaries with operationally simple footprint; pgvector handles embeddings without a separate vector DB.Noisy-neighbour risk in early phases; mitigated by per-DB resource limits, easy split later.
ADR-002Next.js for customer surfaces, React SPA/PWA for operatorsSSR/ISR/SEO matter on customer surfaces; operator surfaces are data-dense and benefit from SPA tooling. Both share the design system.Two build pipelines, but unified by shared packages.
ADR-003Redis Streams now → Kafka laterLower operational cost in early phases; Kafka is adopted only when consumer count or throughput demands it.Eventual migration cost; mitigated by abstracting bus behind a thin client.
ADR-004Temporal for long-running workflowsVisit lifecycles, drip campaigns, retry/escalation logic survive restarts and are inspectable.Operational learning curve; limited to 2–3 workflows in Phase 2.
ADR-005Agents are event handlers, not servicesAgents act through service APIs; humans and agents are interchangeable from the data-plane's view, making everything reversible and auditable.Slightly more plumbing (subscribe + invoke) vs. embedding logic in services.
ADR-006Provider-agnostic AI GatewayAvoids vendor lock-in; centralizes cost, prompt versioning, PII scrubbing.Extra hop adds latency; mitigated by caching and route hints.
ADR-007Single primary region, edge CDNCustomer base concentrated in India in early phases; cost-effective.Multi-region deferred to Phase 7.
ADR-008Shared @kiana/contracts packageOpenAPI-generated TS types; no drift between services and frontends.Requires release discipline; one package version per release train.
